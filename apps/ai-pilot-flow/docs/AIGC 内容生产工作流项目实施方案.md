# ai-pilot-flow AIGC 内容生产工作流项目实施方案

## 1. 项目概述

`ai-pilot-flow` 定位为一个面向企业内容团队的 `AIGC 内容生产工作流 SaaS 平台`，第一阶段重点解决内容生产过程中的标准化、批量化、审核协同和交付追踪问题。

项目目标不是一开始就做成通用 AI 平台，而是先围绕真实内容生产链路，建设一套可交付、可追溯、可扩展的工作流系统。

## 2. 项目定位

### 2.1 产品形态

- 面向内容生产场景的 AIGC 工作流平台
- 以 SaaS 方式交付
- 优先支撑营销、运营、新媒体、电商内容团队

### 2.2 一期核心方向

- 以 `工作流编排` 作为第一优先级
- 以 `稳定产品化` 作为技术路线
- 聚焦“素材输入 -> AI 处理 -> 人工审核 -> 内容导出”的闭环

### 2.3 一期成功标准

- 用户可通过可视化工作流搭建内容生产流程
- 一条工作流可稳定完成多步骤内容生成
- 支持人工审核和退回修改
- 支持批量执行和结果导出
- 支持完整的执行日志与节点追溯

## 3. 业务目标

### 3.1 核心业务价值

- 将分散的内容生产动作沉淀为标准流程
- 降低人工重复创作成本
- 提升内容批量生产效率
- 保证品牌内容风格一致性
- 建立可审核、可复盘的内容交付链路

### 3.2 典型适用场景

- 商品卖点提炼与批量文案生成
- 社媒内容批量生成
- 专题页文案生成与多版本改写
- SEO 内容生产
- 图文脚本、视频口播稿生成
- 品牌活动内容加工与审核

## 4. 一期范围

### 4.1 一期建议纳入

- 可视化工作流编辑器
- 工作流版本管理
- 多步骤节点执行引擎
- Prompt 模板与变量系统
- 文本类内容生成能力
- 批量任务执行
- 人工审核节点
- 执行日志与失败重试
- 内容结果导出
- 基础租户、用户、角色权限

### 4.2 一期不建议纳入

- 复杂多 Agent 自主协作
- 插件市场和开放生态
- 重型知识图谱系统
- 复杂计费中心
- 大规模行业集成平台

## 5. 产品能力设计

### 5.1 工作流编排能力

- 支持节点拖拽编排
- 支持条件分支
- 支持循环处理
- 支持上下文变量传递
- 支持失败重试和超时控制
- 支持人工审核与人工介入

### 5.2 内容生产能力

- Prompt 模板节点
- LLM 生成节点
- 内容改写节点
- 内容扩写节点
- 摘要提炼节点
- 结构化提取节点
- 文本格式转换节点

### 5.3 素材与输入能力

- 支持文本输入
- 支持表格批量导入
- 支持 URL 内容提取
- 支持图片类扩展入口
- 后续支持 PDF、音视频类素材

### 5.4 协作与交付能力

- 草稿状态管理
- 审核通过与退回
- 内容版本留痕
- 结果导出为 Markdown、CSV 等格式
- 后续扩展第三方内容平台发布

## 6. 用户角色设计

### 6.1 角色建议

- `Owner`：租户拥有者，管理全局资源和配置
- `Admin`：管理工作流、成员和资产
- `Editor`：编辑工作流和执行内容任务
- `Reviewer`：审核与退回内容
- `Viewer`：只读查看结果和日志

### 6.2 角色目标

- 运营人员关注批量生成和快速交付
- 内容编辑关注内容质量和可修改性
- 审核人员关注流程把控和合规
- 管理人员关注模板复用、成本和效率

## 7. 工作流引擎设计

### 7.1 设计原则

- 工作流定义与执行逻辑解耦
- 工作流执行与页面渲染解耦
- 支持版本发布与历史追溯
- 支持异步执行与失败恢复

### 7.2 节点类型建议

- `Start`：工作流启动与输入参数定义
- `Prompt`：模板拼装与变量注入
- `LLM`：模型调用与结果返回
- `Transform`：数据清洗与格式转换
- `Condition`：条件判断与分支路由
- `Loop`：批量数据遍历
- `HumanReview`：人工审核和编辑
- `Export`：导出文件或推送外部系统
- `End`：结果聚合和流程结束

### 7.3 执行模型建议

- 采用 `DAG + 状态机` 方式执行
- 每次运行生成唯一 `run_id`
- 每个节点执行生成唯一 `node_run_id`
- 支持节点级重试、超时、取消和断点续跑

### 7.4 关键能力

- 幂等控制
- 重试机制
- 超时熔断
- 并发限制
- 执行日志
- 审计追踪

## 8. AI 能力层设计

### 8.1 统一模型网关

不建议业务层直接调用模型 API，建议建设统一 `LLM Gateway`，统一管理不同模型供应商的接入、参数、降级和成本统计。

### 8.2 模型网关职责

- 统一接入 OpenAI、Claude、Gemini、国产模型和私有模型
- 标准化模型请求参数
- 管理超时、重试和限流
- 支持主备模型切换与降级
- 统计 Token 消耗和调用成本
- 记录模型调用日志
- 做输出格式校验与安全拦截

### 8.3 一期模型策略

- 1 个主模型负责高质量生成
- 1 个辅模型负责改写、摘要、分类等低成本任务
- 1 个备用模型负责故障兜底

### 8.4 输出策略

- 优先输出文本结果
- 同时支持结构化 JSON 输出
- 对结构化结果进行 Schema 校验

## 9. 系统架构建议

### 9.1 推荐技术栈

- 前端：`Next.js + React + TypeScript`
- UI：工作台业务组件 + 可视化编排画布
- 工作流编辑器：`React Flow`
- API / BFF：`Next.js Route Handlers` 或独立 `NestJS`
- 队列：`BullMQ + Redis`
- 数据库：`PostgreSQL`
- 文件存储：`S3 / OSS / COS`
- 监控：`OpenTelemetry + Grafana`

### 9.2 架构分层

#### 展示层

- 控制台
- 工作流编辑器
- 任务中心
- 审核台
- 模板中心

#### 应用层

- 工作流管理
- 任务管理
- 资产管理
- 审核协作
- 租户与权限管理

#### 编排层

- 工作流 DSL
- 节点定义
- 变量上下文
- 执行计划
- 版本发布

#### 执行层

- 调度器
- 队列消费
- 幂等处理
- 重试机制
- 超时管理

#### AI 能力层

- Prompt 模板
- 模型路由
- 输出校验
- 风险拦截
- 成本统计

#### 基础设施层

- PostgreSQL
- Redis
- 对象存储
- 日志
- 监控
- 审计

## 10. 数据设计建议

### 10.1 核心实体

- `tenant`
- `user`
- `workspace`
- `workflow`
- `workflow_version`
- `workflow_run`
- `node_run`
- `asset`
- `prompt_template`
- `brand_profile`
- `review_task`
- `usage_record`
- `provider_config`

### 10.2 关键说明

- 所有核心数据建议带 `tenant_id`
- 工作流必须支持版本发布
- 运行记录要与发布版本绑定
- 节点执行记录需要保存输入、输出、耗时和状态
- 成本统计需要能追溯到租户、工作流、节点和模型

## 11. 多租户与权限

### 11.1 多租户策略

- 一期建议采用逻辑多租户
- 所有业务表通过 `tenant_id` 做数据隔离
- 权限校验统一在应用层与数据访问层处理

### 11.2 审计要求

至少需要记录以下行为：

- 谁创建和发布了工作流
- 谁修改了 Prompt 模板
- 谁执行了批量任务
- 谁审核通过或退回了内容
- 哪个模型生成了最终结果

## 12. 稳定性设计

### 12.1 稳定性原则

- 关键链路异步化
- 任务执行可重试
- 模型调用可降级
- 节点执行可恢复
- 任务结果可回放

### 12.2 需要重点处理的问题

- 同一任务重复触发的幂等问题
- 模型超时、429、网络异常的自动重试
- 主模型不可用时的备用模型切换
- 批量任务的并发限流
- 大规模执行时的成本控制

## 13. 安全与风控

### 13.1 基础安全

- API 限流
- 租户级配额控制
- 文件访问权限隔离
- 鉴权和会话安全

### 13.2 AIGC 风险控制

- 敏感词过滤
- Prompt 注入防护
- 输出内容安全检测
- 结构化结果校验
- 高风险任务的人审兜底

## 14. MVP 范围建议

第一版建议覆盖以下最小闭环：

- 工作流编辑器
- 6 到 8 个核心节点
- 至少 2 个模型供应商接入
- 批量内容生成
- 人工审核节点
- 执行日志与结果回放
- 模板库
- 基础租户与权限
- 结果导出
- 基础用量统计

## 15. 分阶段实施建议

### 15.1 阶段 0：需求建模

- 明确 3 个高价值内容场景
- 明确用户角色和流程节点
- 明确输入物、输出物和审核节点
- 明确一线业务团队的真实使用方式

### 15.2 阶段 1：MVP 开发

- 搭建工作流编辑器
- 实现工作流 DSL 和执行引擎
- 建设统一模型网关
- 实现批量任务、审核流、导出能力
- 完成基础权限和租户能力

### 15.3 阶段 2：产品化增强

- 增加版本管理和模板中心
- 增加配额、成本、告警能力
- 增加系统监控和审计
- 增加更多外部系统连接器

### 15.4 阶段 3：智能化增强

- 引入品牌知识库与风格库
- 引入内容质量评分
- 引入自动优化策略
- 引入更强的 Agent 协作能力

## 16. 团队配置建议

- 1 名产品 / 解决方案负责人
- 1 名前端主程
- 1 名后端 / 架构负责人
- 1 名 AI 应用工程师
- 1 名测试 / 实施工程师

最小启动团队可压缩为：

- 2 名全栈工程师
- 1 名 AI / 后端工程师

## 17. 项目风险

### 17.1 主要风险

- 需求范围快速膨胀，偏离内容生产主场景
- 模型调用成本随批量任务快速上升
- 工作流执行链路复杂，稳定性不足
- 缺乏审核与内容治理导致交付质量波动

### 17.2 风险应对

- 明确一期范围，不做通用平台化扩张
- 建立租户、任务、节点多维成本统计
- 对关键节点做可重试、可恢复设计
- 对关键内容输出增加人工审核兜底

## 18. 结论与建议

对于 `ai-pilot-flow`，建议第一阶段明确定位为：

- `AIGC 内容生产工作流 SaaS`

建议的一期建设主骨架为：

- `工作流引擎`
- `模型网关`
- `审核协作`
- `批量执行`
- `日志追踪`

建议的一期实施目标为：

- 先让内容生产工作流可配置、可运行、可审核、可追溯
- 再逐步增强知识库、自动发布、Agent 和行业化能力
